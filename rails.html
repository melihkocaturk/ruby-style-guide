Ruby on Rails Stil Rehberi

Giriş

Bu rehber, gerçek dünyadaki Rails programcılarının diğer gerçek dünyadaki Rails programcıları tarafından sürdürülebilen kod yazabilmesi için en iyi uygulamaları önerir. Bir stil rehberi gerçek dünyadaki kullanımı yansıtır, ne kadar iyi olduğuna bakılmaksızın oluşabilecek riskleri azaltmak için rehber gereklidir.

Rehberde, ilgili kurallar çeşitli bölümlere ayrılmıştır. Kuralların arkasındaki mantığı eklemeye çalıştım (Eğer atlamışsam, bunun oldukça açık olduğunu varsaymışımdır).

Hiçbir yerde kuralların tamamını bulamadım - çoğu profesyonel yazılım mühendisliği tecrübeme, Rails topluluğunun üyelerinden gelen geri bildirim ve önerilere ve çeşitli kabul görmüş Rails programlama kaynaklarına dayanıyor.

Bu rehberin amacı, Ruby on Rails'in geliştirilmesine yönelik bir dizi en iyi uygulama ve stil yönergeleri sunmaktır. Topluluğun katkısıyla hazırlanan Ruby Stil Rehberi için tamamlayıcı bir rehberdir.

Yapılandırma (Configuration)

  • Özel başlatma kodlarını config/initializers içine yerleştirin. initializers içindeki kodlar uygulama başlatılırken yürütülür.

  • Her gem için başlatma kodunu, gem ile aynı ada sahip ayrı bir dosyada tutun, örneğin carrierwave.rb, active_admin.rb, vb.

  • Geliştirme (development), test ve üretim (production) ortamlarını doğru şekilde yapılandırın (config/environments/ altındaki ilgili dosyalar).

    • Ön derleme için ek varlıkları belirtin (varsa):

      # config/environments/production.rb
# Precompile additional assets (application.js, application.css,
#and all non-JS/CSS are already added)
config.assets.precompile += %w( rails_admin/rails_admin.css rails_admin/rails_admin.js )

  • config/application.rb dosyasında tüm ortamlar için geçerli olan yapılandırmayı saklayın.

  • production ortamına çok benzeyen ek bir staging ortamı oluşturun.

  • Tüm konfigürasyonları config/ dizini altındaki YAML dosyalarında saklayın. Rails 4.2'den bu yana YAML konfigürasyon dosyaları, config_for metodu ile kolayca yüklenebilir:

    Rails::Application.config_for(:yaml_file)

Yönlendirme (Routing)

  • RESTful bir kaynağa daha fazla eylem (action) eklemeniz gerektiğinde (gerçekten ihtiyacınız var mı?) member ve collection rotalarını kullanın.

    # bad
get 'subscriptions/:id/unsubscribe'
resources :subscriptions

# good
resources :subscriptions do
  get 'unsubscribe', on: :member
end

# bad
get 'photos/search'
resources :photos

# good
resources :photos do
  get 'search', on: :collection
end

  • Birden fazla member/collection tanımlamanız gerekiyorsa, alternatif blok sözdizimini kullanın.

    resources :subscriptions do
  member do
    get 'unsubscribe'
    # more routes
  end
end

resources :photos do
  collection do
    get 'search'
    # more routes
  end
end

  • ActiveRecord modelleri arasındaki ilişkiyi daha iyi ifade etmek için rotaları iç içe kullanın.

    class Post < ActiveRecord::Base
  has_many :comments
end

class Comment < ActiveRecord::Base
  belongs_to :post
end

# routes.rb
resources :posts do
  resources :comments
end

  • Rotaları 1 seviyeden daha derinlere yerleştirmeniz gerekirse, shallow: true seçeneğini kullanın. Bu, kullanıcıyı uzun URL'lerden posts/1/comments/5/versions/7/edit ve sizi uzun URL yardımcılarından edit_post_comment_version kurtarır.

    resources :posts, shallow: true do
  resources :comments do
    resources :versions
  end
end

  • İlgili rotaları gruplandırmak için ad alanlarını (namespace) kullanın.

    namespace :admin do
  # Directs /admin/products/* to Admin::ProductsController
  # (app/controllers/admin/products_controller.rb)
  resources :products
end

  • Asla eski dağınık denetleyici (controller) rotalarını kullanmayın. Bu rota her bir denetleyicideki tüm eylemleri GET ile ulaşılabilir yapacaktır.

    # very bad
match ':controller(/:action(/:id(.:format)))'

  • Herhangi bir rota tanımlamak için match kullanmayın.

Denetleyiciler (Controllers)

  • Denetleyicileri zayıf tutun - yalnızca görüntü (view) katmanı için veri almalı ve herhangi bir iş mantığı içermemelidir (tüm iş mantığı doğal olarak modelde bulunmalıdır).

  • Her denetleyici eylemi (ideal olarak) başlangıçtaki find veya new dışında yalnızca bir yöntemi çağırmalıdır.

  • Denetleyici ve görünüm arasında en fazla iki örnek değişkeni paylaşın.

  • ActionFilter seçeneğinde belirtilen denetleyici eylemleri sözcüksel kapsamda olmalıdır. Miras alınan bir işlem için belirtilen ActionFilter, bu eylem üzerindeki etkisinin kapsamını anlamayı zorlaştırır.

    # bad
class UsersController < ApplicationController
  before_action :require_login, only: :export
end

# good
class UsersController < ApplicationController
  before_action :require_login, only: :export

  def export
  end
end

Görselleştirme (Rendering)

  • Satır içinde görüntü oluşturmak yerine şablon kullanmayı tercih edin.

    # very bad
class ProductsController < ApplicationController
  def index
    render inline: "<% products.each do |p| %>
    <%= p.name %>
    <% end %>", type: :erb
  end
end

# good
## app/views/products/index.html.erb
<%= render partial: 'product', collection: products %>

## app/views/products/_product.html.erb

    <%= product.name %>
    

    <%= product.price %>
    

## app/controllers/foo_controller.rb
class ProductsController < ApplicationController
  def index
    render :index
  end
end

  • render text: yerine render plain: tercih edin.

    # bad - sets MIME type to `text/html`
...
render text: 'Ruby!'
...

# bad - requires explicit MIME type declaration
...
render text: 'Ruby!', content_type: 'text/plain'
...

# good - short and precise
...
render plain: 'Ruby!'
...

  • Sayısal HTTP durum kodlarına karşılık gelen sembolleri tercih edin. Anlamlılar ve daha az bilinen HTTP durum kodları için "sihirli" sayılar gibi görünmüyorlar.

    # bad
...
render status: 403
...

# good
...
render status: :forbidden

Modeller

  • ActiveRecord olmayan model sınıflarını özgürce oluşturabilirsiniz.

  • Modelleri kısaltma kullanmadan anlamlı (fakat kısa) isimlerle adlandırın.

  • ActiveRecord veritabanı işlevi olmadan ActiveRecord davranışını (doğrulama gibi) destekleyen model nesnelerine ihtiyacınız varsa, ActiveAttr gem'i kullanın.

    class Message
  include ActiveAttr::Model

  attribute :name
  attribute :email
  attribute :content
  attribute :priority

  attr_accessible :name, :email, :content

  validates :name, presence: true
  validates :email, format: { with: /\A[-a-z0-9_+\.]+\@([-a-z0-9]+\.)+[a-z0-9]{2,4}\z/i }
  validates :content, length: { maximum: 500 }
end

    Konuyla ilgili daha kapsamlı örnekler için RailsCast'e bakın.

  • İş mantığı için anlamlı değilse, modelinize yalnızca verilerinizi biçimlendiren yöntemler (HTML üreten kod gibi) yerleştirmeyin. Bu yöntemlerin büyük olasılıkla yalnızca görünüm katmanından çağrılması gerekir, bu yüzden onların yeri yardımcılar (helpers)'dır. Modellerinizi yalnızca iş mantığı ve veri sürekliliği için saklayın.

ActiveRecord

  • Çok iyi bir nedeniniz olmadıkça (kontrolünüz altında olmayan bir veritabanı gibi) ActiveRecord varsayılanlarını (tablo adları, birincil anahtar vb.) değiştirmekten kaçının.

    # bad - don't do this if you can modify the schema
class Transaction < ActiveRecord::Base
  self.table_name = 'order'
  ...
end

  • Makro tarzı yöntemleri (has_many, validates, vb.) sınıf tanımının başında gruplandırın.

    class User < ActiveRecord::Base
  # keep the default scope first (if any)
  default_scope { where(active: true) }

  # constants come up next
  COLORS = %w(red green blue)

  # afterwards we put attr related macros
  attr_accessor :formatted_date_of_birth

  attr_accessible :login, :first_name, :last_name, :email, :password

  # Rails4+ enums after attr macros, prefer the hash syntax
  enum gender: { female: 0, male: 1 }

  # followed by association macros
  belongs_to :country

  has_many :authentications, dependent: :destroy

  # and validation macros
  validates :email, presence: true
  validates :username, presence: true
  validates :username, uniqueness: { case_sensitive: false }
  validates :username, format: { with: /\A[A-Za-z][A-Za-z0-9._-]{2,19}\z/ }
  validates :password, format: { with: /\A\S{8,128}\z/, allow_nil: true }

  # next we have callbacks
  before_save :cook
  before_save :update_username_lower

  # other macros (like devise's) should be placed after the callbacks

  ...
end

  • has_and_belongs_to_many yerine has_many :through tercih edin. has_many :through kullanmak, birleştirme modelinde ek özellikler ve doğrulamalar sağlar.

    # not so good - using has_and_belongs_to_many
class User < ActiveRecord::Base
  has_and_belongs_to_many :groups
end

class Group < ActiveRecord::Base
  has_and_belongs_to_many :users
end

# preferred way - using has_many :through
class User < ActiveRecord::Base
  has_many :memberships
  has_many :groups, through: :memberships
end

class Membership < ActiveRecord::Base
  belongs_to :user
  belongs_to :group
end

class Group < ActiveRecord::Base
  has_many :memberships
  has_many :users, through: :memberships
end

  • read_attribute(:attribute) yerine self[:attribute] tercih edin.

    # bad
def amount
  read_attribute(:amount) * 100
end

# good
def amount
  self[:amount] * 100
end

  • write_attribute(:attribute, value) yerine self[:attribute] = value tercih edin.

    # bad
def amount
  write_attribute(:amount, 100)
end

# good
def amount
  self[:amount] = 100
end

  • Her zaman yeni doğrulama stilini kullanın.

    # bad
validates_presence_of :email
validates_length_of :email, maximum: 100

# good
validates :email, presence: true, length: { maximum: 100 }

  • Doğrulamaları okumayı kolaylaştırmak için doğrulama başına birden çok özellik listelemeyin.

    # bad
validates :email, :password, presence: true
validates :email, length: { maximum: 100 }

# good
validates :email, presence: true, length: { maximum: 100 }
validates :password, presence: true

  • Özel bir doğrulama bir kereden fazla kullanıldığında ya da düzenli ifadeler ile eşleştirme yapıyorsanız, özel bir validator dosyası yaratın.

    # bad
class Person
  validates :email, format: { with: /\A([^@\s]+)@((?:[-a-z0-9]+\.)+[a-z]{2,})\z/i }
end

# good
class EmailValidator < ActiveModel::EachValidator
  def validate_each(record, attribute, value)
    record.errors[attribute] << (options[:message] || 'is not a valid email') unless value =~ /\A([^@\s]+)@((?:[-a-z0-9]+\.)+[a-z]{2,})\z/i
  end
end

class Person
  validates :email, email: true
end

  • Özel doğrulayıcıları app/validators altında tutun.

  • Adlandırılmış kapsamları (scope) özgürce kullanın.

    class User < ActiveRecord::Base
  scope :active, -> { where(active: true) }
  scope :inactive, -> { where(active: false) }

  scope :with_orders, -> { joins(:orders).select('distinct(users.id)') }
end

  • Lambda ile kapsam tanımlandığında ve parametreler çok karmaşık hale geldiğinde, bunun yerine aynı amaca hizmet eden ve ActiveRecord::Relation nesnesi döndüren bir sınıf yöntemi tanımlamak tercih edilir. Muhtemelen bunun gibi daha basit kapsamları tanımlayabilirsiniz.

    class User < ActiveRecord::Base
  def self.with_orders
    joins(:orders).select('distinct(users.id)')
  end
end

  • Geri çağrıları, uygulanacakları sıraya göre yerleştirin. Başvuru için bkz. Available Callbacks

    #bad
class Person
  after_commit/after_rollback :after_commit_callback
  after_save :after_save_callback
  around_save :around_save_callback
  after_update :after_update_callback
  before_update :before_update_callback
  after_validation :after_validation_callback
  before_validation :before_validation_callback
  before_save :before_save_callback
  before_create :before_create_callback
  after_create :after_create_callback
  around_create :around_create_callback
  around_update :around_update_callback
end

#good
class Person
  before_validation :before_validation_callback
  after_validation :after_validation_callback
  before_save :before_save_callback
  around_save :around_save_callback
  before_create :before_create_callback
  around_create :around_create_callback
  after_create :after_create_callback
  before_update :before_update_callback
  around_update :around_update_callback
  after_update :after_update_callback
  after_save :after_save_callback
  after_commit/after_rollback :after_commit_callback
end

  • Aşağıdaki yöntemlerin davranışına dikkat edin. Model doğrulamalarını çalıştırmazlar ve modelin durumunu kolayca bozabilirler.

    # bad
Article.first.decrement!(:view_count)
DiscussionBoard.decrement_counter(:post_count, 5)
Article.first.increment!(:view_count)
DiscussionBoard.increment_counter(:post_count, 5)
person.toggle :active
product.touch
Billing.update_all("category = 'authorized', author = 'David'")
user.update_attribute(:website, 'example.com')
user.update_columns(last_request_at: Time.current)
Post.update_counters 5, comment_count: -1, action_count: 1

# good
user.update_attributes(website: 'example.com')

  • Kullanıcı dostu URL’leri kullanın. Modelin açıklayıcı bir özelliğini id yerine gösterin. Bunu yapmanın birden fazla yolu vardır:

    • Modelin to_param yönteminin üzerine yazın. Bu yöntem, Rails tarafından nesneye URL oluşturmak için kullanılır. Varsayılan uygulama kaydın id'sini bir dize olarak döndürür. İnsan tarafından okunabilen başka bir özellik eklenerek geçersiz kılınabilir.

      class Person
  def to_param
    "#{id} #{name}".parameterize
  end
end

    Bunu URL dostu bir değere dönüştürmek için parameterize ifadesi kullanılır. Nesnenin id'si başta olmalıdır, böylece ActiveRecord'un find yöntemiyle bulunabilir.

    • friendly_id gemini kullanın. Model id'si yerine açıklayıcı bir özelliğini kullanarak insan tarafından okunabilen URL'lerin oluşturulmasını sağlar.

      class Person
  extend FriendlyId
  friendly_id :name, use: :slugged
end

    Kullanımı hakkında daha fazla bilgi için gem dökümantasyonuna bakın.

  • ActiveRecord nesneleri üzerinde yineleme yapmak için find_each komutunu kullanın. Veritabanında koleksiyonlar aracılığıyla yineleme yapmak (örneğin, all yöntemini kullanarak), tüm nesneleri bir seferde başlatmaya çalışacağı için oldukça verimsizdir. Toplu işlem yöntemleri, kayıtlarla gruplar halinde çalışmanıza izin vererek bellek tüketimini büyük ölçüde azaltır.

    # bad
Person.all.each do |person|
  person.do_awesome_stuff
end

Person.where('age > 21').each do |person|
  person.party_all_night!
end

# good
Person.find_each do |person|
  person.do_awesome_stuff
end

Person.where('age > 21').find_each do |person|
  person.party_all_night!
end

  • before_destroy geri çağrılarını kullanırken her zaman prepend: true ile doğrulama yapın.

    # bad (roles will be deleted automatically even if super_admin? is true)
has_many :roles, dependent: :destroy

before_destroy :ensure_deletable

def ensure_deletable
  raise "Cannot delete super admin." if super_admin?
end

# good
has_many :roles, dependent: :destroy

before_destroy :ensure_deletable, prepend: true

def ensure_deletable
  raise "Cannot delete super admin." if super_admin?
end

  • has_many ve has_one ilişkilerinde dependent seçeneğini tanımlayın.

    # bad
class Post < ActiveRecord::Base
  has_many :comments
end

# good
class Post < ActiveRecord::Base
  has_many :comments, dependent: :destroy
end

  • Kalıcı ActiveRecord nesnelerinde daima istisna fırlatan bang! yöntemini kullanın veya yöntem dönüş değerini işleyin. 

    # bad
user.create(name: 'Bruce')

# bad
user.save

# good
user.create!(name: 'Bruce')
# or
bruce = user.create(name: 'Bruce')
if bruce.persisted?
  ...
else
  ...
end

# good
user.save!
# or
if user.save
  ...
else
  ...
end

ActiveRecord Queries

  • Kodunuzu SQL enjeksiyon saldırılarına duyarlı hale getireceğinden sorgu içinde string interpolasyonundan kaçının.

    # bad - param will be interpolated unescaped
Client.where("orders_count = #{params[:orders]}")

# good - param will be properly escaped
Client.where('orders_count = ?', params[:orders])

  • Birincil anahtar ile tek bir kayıt almanız ve kayıt bulunmadığında ActiveRecord::RecordNotFound istisnasını fırlatmanız gerektiğinde where.take!, find_by! veya find_by_id! yerine find kullanın.

    # bad
User.where(id: id).take!

# bad
User.find_by_id!(id)

# bad
User.find_by!(id: id)

# good
User.find(id)

  • Bir veya daha fazla nitelik ile tek bir kayıt almanız ve kayıt bulunmadığında nil döndürmeniz gerektiğinde where.take veya find_by_attribute yerine find_by kullanın.

    # bad
User.where(email: email).take
User.where(first_name: 'Bruce', last_name: 'Wayne').take

# bad
User.find_by_email(email)
# bad, deprecated in ActiveRecord 4.0, removed in 4.1+
User.find_by_first_name_and_last_name('Bruce', 'Wayne')

# good
User.find_by(email: email)
User.find_by(first_name: 'Bruce', last_name: 'Wayne')

  • SQL yerine where.not kullanmayı tercih edin.

    # bad
User.where("id != ?", id)

# good
User.where.not(id: id)

  • Sıralamak için id sütununu kullanmayın. id dizilerinin - çoğu zaman kronolojik olmasına rağmen - belirli bir düzende olması garanti edilmez. Kronolojik olarak sıralamak için zaman damgası sütunu kullanın.

    # bad
scope :chronological, -> { order(id: :asc) }

# good
scope :chronological, -> { order(created_at: :asc) }

  • pluck(:id) yerine ids kullanın.

    # bad
User.pluck(:id)

# good
User.ids

  • find_by_sql gibi bir yöntemde açık bir sorgu belirtirken, squish ile heredocs kullanın. Böylece, birçok araçta (GitHub, Atom ve RubyMine dahil) sözdizimini vurgularken, SQL'i satır başları ve girintilerle okunaklı şekilde biçimlendirebilirsiniz.

    User.find_by_sql(<<-SQL.squish)
  SELECT
    users.id, accounts.plan
  FROM
    users
  INNER JOIN
    accounts
  ON
    accounts.user_id = users.id
  # further complexities...
SQL

    String#squish, girinti ve yeni satır karakterlerini kaldırır, böylece sunucu günlüğünüz bunun gibi bir şey yerine akıcı bir SQL dizesi gösterir.

    SELECT\n    users.id, accounts.plan\n  FROM\n    users\n  INNER JOIN\n    acounts\n  ON\n    accounts.user_id = users.id

  • ActiveRecord nesnelerini sorgularken count (saymak için her zaman bir veritabanı sorgusu oluşturur) yerine size (koleksiyonun yüklü olup olmamasına bağlı olarak count/length davranışı arasında seçim yapar) yada length (her zaman tüm koleksiyonu yükler ve dizi öğelerini sayar) tercih edin.

    # bad
User.count

# good
User.all.size

# good - if you really need to load all users into memory
User.all.length

Taşıma İşlemleri (Migrations)

  • schema.rb (veya structure.sql) sürümünü kontrol altında tutun.

  • Boş bir veritabanı oluşturmak için rake db:migrate yerine db:schema:load kullanın.

  • Varsayılan değerleri uygulama katmanında değil, migration üzerinde uygulayın.

    # bad - application enforced default value
class Product < ActiveRecord::Base
  def amount
    self[:amount] || 0
  end
end

# good - database enforced
class AddDefaultAmountToProducts < ActiveRecord::Migration
  def change
    change_column_default :products, :amount, 0
  end
end

    Tablo varsayılanlarını uygulamak birçok Rails geliştiricisi tarafından önerilmiş olsa da, verilerinizi birçok uygulama hatalarına karşı savunmasız bırakan son derece kırılgan bir yaklaşımdır. Ve çoğu uygulamanın veritabanını diğer uygulamalarla paylaştığı gerçeğini göz önünde bulundurmanız gerekir, bu nedenle Rails uygulamasından veri bütünlüğü sağlamak imkansızdır.

  • Yabancı anahtar kısıtlamalarını uygulayın. Rails 4.2'den itibaren, ActiveRecord yerel olarak yabancı anahtar kısıtlamalarını destekler.

  • Yapısal migration yazarken (tablo veya sütun eklerken) up ve down yerine change kullanın.

    # the old way
class AddNameToPeople < ActiveRecord::Migration
  def up
    add_column :people, :name, :string
  end

  def down
    remove_column :people, :name
  end
end

# the new preferred way
class AddNameToPeople < ActiveRecord::Migration
  def change
    add_column :people, :name, :string
  end
end

  • Modelleri migration içinde kullanmanız gerekiyorsa daha önce bunları tanımladığınızdan emin olun, böylece gelecekte kırık migration ile karşılaşmayacaksınız. 

    # db/migrate/<migration_file_name>.rb
# frozen_string_literal: true

# bad
class ModifyDefaultStatusForProducts < ActiveRecord::Migration
  def change
    old_status = 'pending_manual_approval'
    new_status = 'pending_approval'

    reversible do |dir|
      dir.up do
        Product.where(status: old_status).update_all(status: new_status)
        change_column :products, :status, :string, default: new_status
      end

      dir.down do
        Product.where(status: new_status).update_all(status: old_status)
        change_column :products, :status, :string, default: old_status
      end
    end
  end
end

# good
# Define `table_name` in a custom named class to make sure that
# you run on the same table you had during the creation of the migration.
# In future if you override the `Product` class
# and change the `table_name`, it won't break
# the migration or cause serious data corruption.
class MigrationProduct < ActiveRecord::Base
  self.table_name = :products
end

class ModifyDefaultStatusForProducts < ActiveRecord::Migration
  def change
    old_status = 'pending_manual_approval'
    new_status = 'pending_approval'

    reversible do |dir|
      dir.up do
        MigrationProduct.where(status: old_status).update_all(status: new_status)
        change_column :products, :status, :string, default: new_status
      end

      dir.down do
        MigrationProduct.where(status: new_status).update_all(status: old_status)
        change_column :products, :status, :string, default: old_status
      end
    end
  end
end

  • Rails tarafından otomatik olarak oluşturulan yabancı anahtar adlarına güvenmek yerine, yabancı anahtarlarınızı açıkça adlandırın. (https://guides.rubyonrails.org/active_record_migrations.html#foreign-keys)

    # bad
class AddFkArticlesToAuthors < ActiveRecord::Migration
  def change
    add_foreign_key :articles, :authors
  end
end

# good
class AddFkArticlesToAuthors < ActiveRecord::Migration
  def change
    add_foreign_key :articles, :authors, name: :articles_author_id_fk
  end
end

  • Geri dönüşü olmayan komutları kullanmayın. Tersine çevrilebilir komutlar burada listelenmiştir. 

    # bad
class DropUsers < ActiveRecord::Migration
  def change
    drop_table :users
  end
end

# good
class DropUsers < ActiveRecord::Migration
  def up
    drop_table :users
  end

  def down
    create_table :users do |t|
      t.string :name
    end
  end
end

# good
# In this case, block will be used by create_table in rollback
# http://api.rubyonrails.org/classes/ActiveRecord/ConnectionAdapters.html#method-i-drop_table
class DropUsers < ActiveRecord::Migration
  def change
    drop_table :users do |t|
      t.string :name
    end
  end
end

Görünümler (Views)

  • Model katmanını asla doğrudan görünümde çağırmayın.

  • Görünümde karmaşık biçimlendirmelerden kaçının. Helper metodlar basit durumlar için kullanışlıdır.

  • Kısmi şablon (partial) ve düzen (layout) kullanarak kod tekrarını azaltın.

  • Örnek değişkenleri partial olarak kullanmaktan kaçının, bunun yerine render için yerel bir değişkeni parametre olarak verin. Partial, değişkenin farklı bir isme sahip olduğu veya değişkenin bulunmadığı farklı bir denetleyicide veya eylemde kullanılabilir. Bu gibi durumlarda, tanımsız bir örnek değişken istisna fırlatmaz.

    <!-- bad -->
<!-- app/views/courses/show.html.erb -->
<%= render 'course_description' %>
<!-- app/views/courses/_course_description.html.erb -->
<%= @course.description %>

<!-- good -->
<!-- app/views/courses/show.html.erb -->
<%= render 'course_description', course: @course %>
<!-- app/views/courses/_course_description.html.erb -->
<%= course.description %>

Uluslararasılaştırma (Internationalization)

  • Görünümlerde, modellerde ve denetleyicilerinde hiçbir yerel ayar kullanılmamalıdır. Bu metinler config/locales dizinindeki ilgili dosyalara taşınmalıdır.

  • ActiveRecord model etiketlerinin çevrilmesi gerektiğinde, activerecord kapsamını kullanın:

    en:
  activerecord:
    models:
      user: Member
    attributes:
      user:
        name: 'Full name'

    Devamında User.model_name.human "Member" ve User.human_attribute_name("name") "Full name" döndürür. Özelliklerin çevirileri görünümlerde etiket olarak kullanılacaktır.

  • Görünümlerde kullanılan metinleri ActiveRecord niteliklerinin çevirilerinden ayırın. Modeller için dosyaları locales/models klasörüne ve görünümlerde kullanılan metinleri locales/views klasörüne yerleştirin.

    • Yerel dosyaların düzenlenmesi ek dizinlerle yapıldığında, bu dizinlerin yüklenebilmesi için application.rb dosyasında tanımlanması gerekir.

      # config/application.rb
config.i18n.load_path += Dir[Rails.root.join('config', 'locales', '**', '*.{rb,yml}')]

  • Tarih veya para birimi biçimleri gibi paylaşılan yerelleştirme seçeneklerini locales dizininin altındaki dosyalara yerleştirin.

  • I18n yöntemlerinin kısa biçimlerini kullanın: I18n.translate yerine I18n.t ve I18n.localize yerine I18n.l.

  • Görünümlerde kullanılan metinler için "lazy lookup" kullanın. Diyelim ki şu yapıya sahibiz:

    en:
  users:
    show:
      title: 'User details page'

    users.show.title değeri app/views/users/show.html.haml içinde şu şekilde kullanılabilir:

    = t '.title'

  • :scope yerine nokta ile ayrılmış anahtarlar kullanın.

    # bad
I18n.t :record_invalid, scope: [:activerecord, :errors, :messages]

# good
I18n.t 'activerecord.errors.messages.record_invalid'

  • Rails I18n ile ilgili daha detaylı bilgi Rails Guides'da bulunabilir.

Varlıklar (Assets)

  • Özel stil sayfaları (stylesheets), javascript dosyalarını ve resimleri app/assets altında saklayın.

  • Uygulamanın kapsamına gerçekten uymayan kütüphaneler için lib/assets dizinini kullanın.

  • JQuery ve bootstrap gibi üçüncü parti kodlar vendor/assets içine yerleştirilmelidir.

  • Mümkünse, varlıkların gem verisoyonlarını kullanın (örneğin, jquery-rails, jquery-ui-rails, bootstrap-sass, zurb-foundation).

E-posta Göndericileri (Mailers)

  • E-posta göndericilerini SomethingMailer olarak adlandırın. Mailer soneki olmadan, neyin mailer olduğu ve hangi görünümün bu mailer ile ilgili olduğu hemen anlaşılamaz.

  • Hem HTML hem de düz metin görünüm şablonlarını kullanın.

  • Geliştirme ortamınızda başarısız e-posta tesliminde ortaya çıkan hataları etkinleştirin. Hatalar varsayılan olarak devre dışıdır.

    # config/environments/development.rb

config.action_mailer.raise_delivery_errors = true

  • Geliştirme ortamında Mailcatcher gibi yerel bir SMTP sunucusu kullanın.

    # config/environments/development.rb

config.action_mailer.smtp_settings = {
  address: 'localhost',
  port: 1025,
  # daha fazla ayar
}

  • Ana bilgisayar adı için varsayılan ayarları tanımlayın.

    # config/environments/development.rb
config.action_mailer.default_url_options = { host: "#{local_ip}:3000" }

# config/environments/production.rb
config.action_mailer.default_url_options = { host: 'your_site.com' }

# in your mailer class
default_url_options[:host] = 'your_site.com'

  • from ve adresleri doğru şekilde biçimlendirin. Aşağıdaki formatı kullanın:

    # in your mailer class
default from: 'Your Name <info@your_site.com>'

  • Test ortamınız için e-posta teslim yönteminin test olarak ayarlandığından emin olun.

    # config/environments/test.rb

config.action_mailer.delivery_method = :test

  • Geliştirme ve üretim için teslim yöntemi smtp olmalıdır.

    # config/environments/development.rb, config/environments/production.rb

config.action_mailer.delivery_method = :smtp

  • HTML e-postaları gönderirken - bazı istemcilerin dış stillerle ilgili sorunları olduğundan - tüm stillerin satır içi olması gerekir. Ancak bu, bakımlarını zorlaştırır ve kod tekrarına yol açar. Stilleri dönüştüren ve bunlara karşılık gelen html etiketlerine yerleştiren iki benzer gem vardır: premailer-rails ve roadie.

  • Sayfa yanıtı oluştururken e-posta göndermekten kaçınılmalıdır. Sayfanın yüklenmesinde gecikmelere neden olur ve birden fazla e-posta gönderilirse istek zaman aşımına uğrayabilir. Bunun üstesinden gelmek için sidekiq gem yardımı ile e-posta arka planda gönderilebilir.

Çekirdek Uzantılar (Core Extension)

  • ActiveSupport#try! yerine Ruby 2.3'ün güvenli navigasyon operatörünü &. kullanmayı tercih edin.

    # bad
obj.try! :fly

# good
obj&.fly

  • ActiveSupport takma adları yerine Ruby'nin standart kütüphane yöntemlerini tercih edin.

    # bad
'the day'.starts_with? 'th'
'the day'.ends_with? 'ay'

# good
'the day'.start_with? 'th'
'the day'.end_with? 'ay'

  • Nadiren kullanılan ActiveSupport uzantıları yerine Ruby'nin standart kütüphanesini tercih edin.

    # bad
(1..50).to_a.forty_two
1.in? [1, 2]
'day'.in? 'the day'

# good
(1..50).to_a[41]
[1, 2].include? 1
'the day'.include? 'day'

  • ActiveSupport'un Array#inquiry ve String#inquiry yöntemleri yerine Ruby karşılaştırma işleçlerini kullanın.

    # bad - String#inquiry
ruby = 'two'.inquiry
ruby.two?

# good
ruby = 'two'
ruby == 'two'

# bad - Array#inquiry
pets = %w(cat dog).inquiry
pets.gopher?

# good
pets = %w(cat dog)
pets.include? 'cat'

Zaman

  • Saat diliminizi application.rb'de uygun şekilde yapılandırın.

    config.time_zone = 'Eastern European Time'
# optional - note it can be only :utc or :local (default is :utc)
config.active_record.default_timezone = :local

  • Time.parse kullanmayın.

    # bad
Time.parse('2015-03-02 19:05:37') # => Will assume time string given is in the system's time zone.

# good
Time.zone.parse('2015-03-02 19:05:37') # => Mon, 02 Mar 2015 19:05:37 EET +02:00

  • String#to_time kullanmayın.

    # bad - assumes time string given is in the system's time zone.
'2015-03-02 19:05:37'.to_time

# good
Time.zone.parse('2015-03-02 19:05:37') # => Mon, 02 Mar 2015 19:05:37 EET +02:00

  • Time.now kullanmayın.

    # bad
Time.now # => Returns system time and ignores your configured time zone.

# good
Time.zone.now # => Fri, 12 Mar 2014 22:04:47 EET +02:00
Time.current # Same thing but shorter.

Paket Yöneticisi (Bundler)

  • Gemfile'da sadece geliştirme veya test için kullanılan gemleri uygun şekilde gruplayın.

  • Projelerinizde yalnızca belirli gemleri kullanın. Az bilinen gemleri eklemeyi düşünüyorsanız, önce kaynak kodunu dikkatlice gözden geçirmelisiniz.

  • Farklı işletim sistemleri kullanan birden fazla geliştiriciye sahip projelerde, işletim sistemine özgü gemlerin doğal sonucu olarak Gemfile.lock sürekli değişecektir. OS X'e özgü gemleri darwin grubuna ve Linux'a özgü gemleri linux grubuna ekleyin:

    # Gemfile
group :darwin do
  gem 'rb-fsevent'
  gem 'growl'
end

group :linux do
  gem 'rb-inotify'
end

    Geliştirme ortamına uygun gemleri çağırmak için config/application.rb dosyasına şunu ekleyin:

    platform = RUBY_PLATFORM.match(/(linux|darwin)/)[0].to_sym
Bundler.require(platform)

  • Gemfile.lock dosyasını versiyon kontrolünden çıkarmayın. Bu rastgele oluşturulmuş bir dosya değildir; takım üyeleriniz bundle install komutunu kullandıklarında tüm gem versiyonlarının aynı olmasını sağlar.

Süreçleri Yönetmek
